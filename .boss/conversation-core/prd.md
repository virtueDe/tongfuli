## 摘要

> 下游 Agent 请优先阅读本节；本功能只定义首个真实用户链路，不要把后台治理、多端铺开或外部资料采集混进本轮实现。

- **核心目标**：在 `Web` 主对话页打通“匿名进入 -> 自动创建会话 -> 发送一轮消息 -> 获得角色化流式回答”的真实在线链路，替换当前首页静态骨架。
- **目标用户**：首次访问 Tongfuli Web 站点、希望立即体验《武林外传》角色对话的匿名用户。
- **主路径**：用户打开首页，看到默认角色与模式；输入问题并点击发送；前端若本地无 `sessionId` 则先调用 `POST /api/v1/public/sessions` 创建匿名会话，再调用 `POST /api/v1/public/sessions/{sessionId}/turns/stream` 获取 SSE 回答流；页面实时展示用户消息、回答增量、完成态和错误态。
- **默认体验**：默认 `clientType=web`、`initialMode=canon`、`initialCharacterId=char_baizhantang`，用户无需登录、无需跳转角色广场、无需先做复杂配置。
- **角色化回答要求**：回答必须同时满足“像角色在说话”和“基本不脱离剧中设定”两个条件；本轮只要求可感知的人设口吻，不要求完成依据展开、模式切换、角色切换的全量体验。
- **前后端契约约束**：严格以 [`contracts/public-api.md`](C:\code\virtue\tongfuli\contracts\public-api.md) 为准接入 `创建会话` 与 `发送消息并获取流式回答` 两个公开接口，流式协议使用 SSE 事件 `answer.delta`、`answer.completed`、`answer.metadata`、`answer.error`。
- **本轮范围内**：Web 主对话页接入真实接口、匿名会话创建、单轮问答流式展示、基础请求中/失败态、最小消息列表状态管理。
- **本轮明确不做**：后台治理、不做外部资料采集、不做小程序或多端同时实现、不做最近会话/反馈/依据展开/分享链路、不做角色切换与模式切换的完整交互闭环。

## 需求背景

当前 Web 首页仅有静态主对话页骨架，文案已经表达“后面再接真实问答链路”，但真实用户无法完成一次有效对话。这不是可选优化，而是产品是否成立的第一道门槛。

相较于总 PRD 中的大范围规划，本 feature 只解决一个真问题：先让用户在 Web 首页上真正聊起来。只要首条真实链路未跑通，后台治理、多模式、多端铺开都没有验证价值。

## 目标与非目标

### 产品目标

1. 让匿名用户在 1 个页面内完成第一次真实对话。
2. 让首页从“展示骨架”升级为“可验证角色化价值”的可用产品。
3. 为后续架构代理和开发代理提供稳定边界：先打通最短链路，再在其上增量加角色切换、模式切换、依据展开等能力。

### 非目标

1. 不解决后台内容治理、审核发布、问答诊断、AI 策略工作台。
2. 不解决外部资料抓取、知识管线、发布态快照生产。
3. 不解决小程序接入、原生 App、跨端一致性。
4. 不解决多轮记忆优化、最近会话、反馈、分享、依据展开。

## 用户与场景

### 目标用户

- `Web` 匿名访客：从首页直接体验角色对话，不愿先登录。
- 《武林外传》轻度或重度兴趣用户：第一关注点不是功能丰富，而是“像不像”和“能不能马上聊”。

### 核心场景

用户打开 Web 首页，看见默认角色“白展堂”和默认模式“原剧模式”，在输入框中输入一句问题，例如“老白为什么怕佟掌柜？”。系统在用户无感知前提下创建匿名会话，并流式返回一段带有白展堂口吻的回答。回答结束后，页面保留本轮问答内容，用户可以确认这不是静态演示，而是真实可交互产品。

## 用户链路

### 主链路

1. 用户进入首页。
2. 页面渲染主对话区、默认角色、默认模式、输入框。
3. 用户输入消息并点击发送。
4. 前端检查当前内存或本地状态是否已有 `sessionId`。
5. 若没有，则调用 `POST /api/v1/public/sessions` 创建匿名会话。
6. 会话创建成功后，调用 `POST /api/v1/public/sessions/{sessionId}/turns/stream` 发起本轮问答。
7. 页面先展示用户消息，再消费 SSE 流，持续拼接助手消息。
8. 收到 `answer.completed` 后将该轮消息标记为完成，可继续留在列表中。
9. 若创建会话或流式问答失败，页面给出明确错误提示，并允许用户重新发送。

### 关键状态

- `idle`：尚未发送消息。
- `creating_session`：正在创建匿名会话。
- `streaming_answer`：正在接收 SSE 增量。
- `completed`：本轮回答结束。
- `error`：创建会话失败或回答失败。

## 功能需求

### FR-001 首页主对话页接入真实接口

- **描述**：替换当前静态占位消息，使 [`apps/web/app/page.tsx`](C:\code\virtue\tongfuli\apps\web\app\page.tsx) 成为可真实提问并显示回答的入口页。
- **优先级**：P0
- **验收标准**：
  - 页面首屏可直接输入，不要求登录。
  - 发送后不再停留在静态演示内容，而是展示真实消息流。
  - 默认角色显示为“白展堂”，默认模式显示为“原剧模式”。

### FR-002 匿名会话自动创建

- **描述**：用户第一次发送消息时，如果前端尚未持有会话标识，则自动调用创建会话接口，不增加额外确认步骤。
- **优先级**：P0
- **接口约束**：
  - 方法：`POST /api/v1/public/sessions`
  - 请求体至少包含 `clientType=web`、`initialMode=canon`、`initialCharacterId=char_baizhantang`、`deviceId`
- **验收标准**：
  - 首次发送时自动创建会话。
  - 创建成功后本轮消息继续发送，不要求用户二次点击。
  - 创建失败时页面有错误提示且可重试。

### FR-003 单轮消息流式回答

- **描述**：基于已创建的会话，发送一轮用户输入并消费 SSE 流，实时渲染角色化回答。
- **优先级**：P0
- **接口约束**：
  - 方法：`POST /api/v1/public/sessions/{sessionId}/turns/stream`
  - 请求体至少包含 `input`、`mode=canon`、`actingCharacterId=char_baizhantang`、`showEvidenceHint=false`
  - 需要处理事件：`answer.delta`、`answer.completed`、`answer.metadata`、`answer.error`
- **验收标准**：
  - 用户发送后立即看到自己的消息进入消息列表。
  - 助手消息按流式增量实时更新，不等待整段完成后一次性出现。
  - 收到完成事件后消息状态切为完成，不再继续追加。
  - 收到错误事件后停止流式渲染并展示失败信息。

### FR-004 最小可用消息列表状态

- **描述**：页面至少要维护“用户消息 + 助手消息 + 发送状态”的最小状态模型，保证一次真实对话可见、可理解、可恢复。
- **优先级**：P0
- **验收标准**：
  - 至少支持 1 轮用户消息和 1 轮助手消息的渲染。
  - 发送中、完成、失败三种状态有可区分反馈。
  - 页面刷新后会话是否保留不作为本轮强制要求，但同页内再次发送必须复用已创建的 `sessionId`。

## 非功能要求

### NFR-001 首次体验必须短路径

- 首页不增加登录、角色广场、模式配置等前置步骤。
- 用户进入页面到可以发送消息的操作路径不超过 1 个输入框和 1 个发送动作。

### NFR-002 流式反馈必须可感知

- 用户点击发送后，界面必须立即进入忙碌态，避免误判为无响应。
- 回答输出必须体现流式生成过程，而不是伪流式占位或整段回填。

### NFR-003 失败路径必须可恢复

- 任一接口失败后，用户不需要刷新整页即可重试。
- 错误信息要区分“创建会话失败”和“回答失败”两类，以便开发与后续观测定位。

## 范围定义

### In Scope

- Web 首页主对话页真实接口接入
- 匿名会话创建
- 单轮消息发送与流式回答展示
- 默认角色与模式展示
- 基础加载态、发送态、错误态处理

### Out of Scope

- 后台治理：本轮不做内容治理、审核发布、AI 策略、问答诊断后台
- 外部资料采集：本轮不做抓取、清洗、审核、发布外部资料链路
- 多端同时实现：本轮只做 Web，不同步实现小程序
- 高阶对话能力：本轮不做角色切换、模式切换、依据展开、分享、最近会话、反馈
- 长链路运营能力：本轮不做埋点分析体系、增长承接页、复杂风控与设备指纹工程化

## 依赖与约束

### 上游依赖

- 公共 API 按 [`contracts/public-api.md`](C:\code\virtue\tongfuli\contracts\public-api.md) 提供可调用的创建会话与流式问答接口。
- AI 编排侧能够基于 `char_baizhantang + canon` 返回角色化文本流。

### 实现约束

- 不允许继续使用纯静态占位消息冒充真实回答。
- 不允许定义一套脱离公共契约的新前端协议。
- 不允许为未来多端或后台能力提前引入重型抽象，先满足 Web 最短链路。

## 验收标准

### 产品验收

1. 打开 Web 首页即可直接提问，无登录门槛。
2. 首次发送会自动创建匿名会话，并成功得到一轮角色化回答。
3. 页面能清楚展示默认角色、用户输入、助手流式输出和失败提示。

### 联调验收

1. 创建会话与发送消息严格走公共 API 契约中的 2 个接口。
2. SSE 四类事件可被前端正确消费，其中 `answer.delta` 与 `answer.completed` 为主路径必验项。
3. 同页第二次发送复用首次返回的 `sessionId`，不重复创建新会话。

## 风险

- 如果后端接口返回结构未落地或与公共契约不一致，Web 接入会被阻塞。
- 如果角色化回答质量过弱，虽然链路打通，但产品价值感仍不足。
- 如果前端错误态处理含糊，联调阶段会把“接口失败”和“流式解析失败”混成一类，定位成本高。
