# Tongfuli 关键技术决策记录

## 决策 1：采用模块化单体 + 独立 AI 编排服务 + 独立知识处理平台

### Decision

主业务系统采用模块化单体承载用户、会话、角色、内容、审核、运营与后台聚合逻辑；AI 编排服务与知识处理平台独立部署。

### Rationale

- 新项目首发阶段最怕的是边界未稳定就过早拆成大量微服务。
- AI 编排链路与知识处理链路在性能目标、扩缩容方式和演进速度上都与业务主系统不同。
- 这种拆法能保持业务主线简单，同时把高异步、高不确定性的能力隔离出去。

### Alternatives considered

- 全微服务：扩展性强，但对当前阶段明显过度设计。
- 大一统单体：首发快，但会把 AI 与异步知识处理复杂度直接倒灌进主系统。

## 决策 2：线上主回答使用闭源强模型，离线任务使用开源模型

### Decision

在线最终回答使用闭源强模型，离线抽取、清洗、分类、打标与部分低价值任务使用开源模型。

### Rationale

- 用户直接感知的是线上回答质量，首发阶段效果优先。
- 离线知识处理量大、成本敏感，适合引入开源模型提升长期控制力。
- 通过统一 LLM Gateway 保留供应商切换能力，降低锁定风险。

### Alternatives considered

- 全闭源：上线快，但长期成本和供应商依赖太高。
- 全开源：长期可控，但首发体验和维护成本风险大。
- 双主链路在线并行：复杂度与稳定性成本过高。

## 决策 3：知识系统按 Canonical / Interpretation / Entertainment 三层分治

### Decision

知识内容按事实真源、解释增强、娱乐延展三层治理，不允许直接混用。

### Rationale

- 事实问题必须可追溯，外部梗文不能反客为主。
- 用户要的是好玩，但平台必须知道“哪里是事实，哪里是演绎”。
- 分层后才有办法做模式系统与来源权重控制。

### Alternatives considered

- 所有内容统一索引：实现简单，但很快会污染事实层。
- 只保留剧本：准确性高，但娱乐性和传播性不足。

## 决策 4：深度结构化剧本，而不是只做文本切片

### Decision

从第一阶段开始把剧本解析成角色、场景、台词、事件、关系、梗点和主题等结构化实体。

### Rationale

- 角色化问答与连续追问依赖结构化世界模型，而不是单纯的文本相似度。
- 深度结构化后，才能做关系推理、场景追溯、角色边界和梗解释。
- 结构化结论可以被审核、版本化和回滚。

### Alternatives considered

- 仅做 chunk + 向量检索：实现快，但后续一定返工。
- 先基础结构化后逐步增强：可行，但当前项目明确要求直接按深度结构化目标设计。

## 决策 5：首发双端采用 Web + 小程序

### Decision

Web 作为完整体验主战场，小程序作为分享与回流承接渠道。

### Rationale

- Web 更适合完整对话体验、专题页、SEO 和后台一体化治理。
- 小程序更适合低门槛访问和社交裂变。
- 原生 App 当前阶段投入产出比低，不应优先。

### Alternatives considered

- 仅 Web：实现更简单，但传播效率不足。
- 直接 App：开发与运营成本过高。

## 决策 6：匿名可用，但后台必须强风控

### Decision

前台默认允许匿名体验，但必须配套频控、指纹、风险评分、分层限额和降级策略。

### Rationale

- 增长优先的消费型 AI 产品必须降低试用门槛。
- 匿名流量最容易引发成本失控和刷量攻击。
- 风控分层比“全量强制登录”更符合当前目标。

### Alternatives considered

- 强制登录：控制简单，但增长受阻。
- 完全匿名不限流：短期看起来丝滑，长期一定失控。
