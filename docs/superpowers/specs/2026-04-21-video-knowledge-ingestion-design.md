# 视频知识采集与治理设计

## 1. 目标

为 Tongfuli 增加企业级视频来源治理能力，面向 `B站` 与 `抖音` 公开页面采集《武林外传》相关视频内容，完成摘要、知识抽取、证据留存、审核发布与统一检索接入。

该能力不是独立的一次性爬虫，而是现有知识处理平台中的正式子系统，和现有 `PDF` 剧本知识一起构成统一知识底座。

## 2. 已确认边界

- 采集方式：使用公开页面浏览器自动化，不限定官方 API。
- 产品定位：企业级产品，不按 `MVP` 方案收缩。
- 入库审核：高置信度自动入库，低置信度进入审核队列。
- 知识库形态：保留原始视频知识库存，同时建设统一检索视图。
- 知识颗粒度：除摘要外，还需保存剧情事件、人物关系、设定知识、名台词候选、证据片段、原文摘录、时间点或位置、置信度。
- 采集范围：账号池主驱动，关键词搜索用于补充发现。
- 运行节奏：小时级批处理，默认账号池每 `1` 小时扫描一次。
- 模型层：通过统一模型网关支持多供应商切换。
- 评论区：只作为线索与辅助证据，不直接进入主知识库。
- 对外使用：会用于对外能力，但不展示第三方来源。
- 对外发布方式：只发布 Tongfuli 自有结构化知识卡片与总结，不直接引用第三方原文。
- 平台范围：首期同时支持 `B站` 与 `抖音`。

## 3. 方案选择

### 3.1 推荐方案

在现有 `services/knowledge-pipeline` 内扩展“视频来源治理子系统”，并将浏览器执行层拆成可独立扩缩的 worker。

### 3.2 备选方案

1. 在现有知识管道内扩展视频来源治理能力。
2. 新建独立 `video-crawler-service`，采集完成后再向知识平台推送。
3. 先做离线批处理工程，后续再平台化。

### 3.3 选型结论

采用方案 `1`。原因如下：

- 与仓库当前架构一致，能直接复用知识抽取、审核、发布、索引刷新能力。
- 更适合和剧本知识形成统一发布视图与统一检索。
- 企业级治理能力需要采集、审核、发布在一个可追溯体系内闭环。

不采用方案 `2` 作为首选，是因为会过早引入额外服务边界和同步复杂度。

不采用方案 `3`，是因为后续几乎必然重构，不适合企业级目标。

## 4. 系统边界

建议将视频来源治理拆成以下模块：

### 4.1 Source Registry

负责账号池、关键词池、抓取频率、信任等级、平台配置、登录态引用与限频策略。

### 4.2 Capture Workers

负责浏览器自动化抓取 `B站`、`抖音` 公开页面，采集视频元信息、简介、字幕、作者信息、封面、发布时间、评论线索，并保留原始页面快照与结构化抓取结果。

### 4.3 Extraction Pipeline

负责转写、摘要、剧情事件抽取、人物关系抽取、设定知识抽取、台词候选抽取、证据片段定位与置信度打分。

### 4.4 Review Pipeline

负责高低置信度分流、冲突内容复核、人工审核队列与审核记录。

### 4.5 Knowledge Publishing

负责原始视频知识库存、统一发布视图、发布快照生成、索引刷新与撤回。

### 4.6 Serving / Retrieval

负责让对外问答、搜索、推荐只读取已发布知识快照。虽然对外不展示第三方来源，但系统内部必须完整追溯来源、抓取时间、证据片段、处理版本与审核记录。

## 5. 数据模型

建议首批核心数据对象如下：

### 5.1 `source_account`

记录平台、账号主页、账号类型、可信等级、抓取频率、状态。

### 5.2 `source_video`

记录平台视频唯一标识、作者、标题、简介、发布时间、标签、链接、抓取状态。

### 5.3 `capture_job`

记录发现、抓取、补抓、失败重试、限频退避、封控异常。

### 5.4 `raw_asset`

保存页面快照、结构化抓取结果、字幕、音频转写、封面、评论线索等原始证据。

### 5.5 `evidence_snippet`

将可用证据切成片段，保存时间点、位置、原文摘录、置信度、处理版本。

### 5.6 `knowledge_candidate`

保存候选知识，类型包括剧情事件、人物关系、设定知识、名台词候选、主题总结。

### 5.7 `review_task`

承接自动审核与人工审核，记录命中规则、模型判断、审核人、结论、备注。

### 5.8 `published_knowledge`

保存已发布知识，仅保留 Tongfuli 自有结构化表达，不直接暴露第三方原文。

### 5.9 `unified_search_view`

将剧本知识与视频发布知识合并到统一检索视图，但内部可反查来源链路。

## 6. 处理流程

### 6.1 发现与采集

1. 调度器按小时级扫描账号池。
2. 关键词任务以较低频率执行，用于补充发现。
3. 平台适配器分别抓取 `B站` 与 `抖音` 页面。
4. 抓取结果统一归一化为内部标准格式。

### 6.2 清洗与抽取

1. 优先抽取平台字幕，没有字幕再走 `ASR` 转写。
2. 对文本做切段、清洗、去噪、时间点映射。
3. 模型网关执行摘要、知识抽取、证据定位与置信度评分。
4. 评论区只作为辅助线索，不直接进入主知识候选。

### 6.3 去重与审核

1. 与已有视频知识和剧本知识做去重。
2. 对冲突知识打回复核。
3. 高置信候选自动进入待发布区。
4. 低置信候选进入人工审核工作台。

### 6.4 发布与检索

1. 通过审核的知识进入发布快照。
2. 刷新统一检索视图与相关索引。
3. 对外问答、搜索、推荐只读取发布态知识。
4. 支持按快照回滚与撤回。

## 7. 技术选型与服务拆分

### 7.1 主体技术方向

- 主知识服务：继续使用现有 `Python 3.12 + FastAPI` 的 `knowledge-pipeline`
- 浏览器自动化：`Playwright`
- 工作流：`Temporal`
- 事件流：`Kafka`
- 元数据与审核记录：`PostgreSQL`
- 原始资产：对象存储
- 检索：`OpenSearch + pgvector`
- 图关系能力：后续逐步引入 `Neo4j`

### 7.2 服务职责划分

建议拆成四层职责：

1. `Scheduler`
2. `Capture Runtime`
3. `Knowledge Processing`
4. `Review & Publish`

### 7.3 结构建议

优先在现有仓库中扩展：

- `contracts/`
- `services/knowledge-pipeline/`
- `docs/architecture/`
- `apps/admin/`

如需隔离浏览器运行时，可新增 `services/video-capture/`，但仍由统一知识治理体系调度。

## 8. 风控与可运维性

### 8.1 合规与台账

内部必须记录平台、页面、抓取时间、采集方式、用途、处理链路，满足法务、风控、纠错与争议处理需要。

### 8.2 平台风控防护

- 每个平台、每账号池、每 worker 设置限频和退避。
- 登录态、代理、浏览器指纹按平台隔离。
- 对验证码、跳登录、结构变化、内容缺失、封控等异常分类上报。

### 8.3 数据正确性

- 采集、转写、抽取流程必须保证幂等。
- 同一视频的重复发现、补抓、重跑不能生成重复候选知识。

### 8.4 观测指标

至少需要观测：

- 发现视频数
- 抓取成功率
- 平台封控率
- 转写成功率
- 抽取通过率
- 自动入库率
- 人工审核积压
- 发布延迟
- 每平台与每模型成本

## 9. 分阶段实施路线图

### 9.1 `P0` 基础骨架

目标是让视频来源治理正式挂接到知识平台：

- 扩展导入契约，新增 `sourceType=video`、平台、作者、证据片段、处理版本字段。
- 在 `services/knowledge-pipeline` 下补齐 `capture`、`adapters`、`normalization`、`evidence`、`publish` 模块骨架。
- 建立核心实体：`source_account`、`source_video`、`capture_job`、`raw_asset`、`knowledge_candidate`、`review_task`。
- 打通账号发现到原始资产落盘的最小链路。

### 9.2 `P1` 平台采集能力

目标是接通 `B站` 与 `抖音` 的生产级采集：

- 建立 `Playwright` 采集 runtime
- 完成双平台 adapter
- 支持账号池扫描与关键词补充
- 采集页面快照、字幕、简介、标签、作者信息
- 建立异常分类与重试机制

### 9.3 `P2` 知识抽取与审核

目标是把视频内容转为可治理知识：

- 字幕优先，没有字幕再做 `ASR`
- 模型网关接入多供应商
- 执行摘要、剧情事件、人物关系、设定知识、名台词候选抽取
- 建立证据定位、置信度评分、去重与冲突检测
- 落地自动分流与人工审核

### 9.4 `P3` 发布与统一检索

目标是让视频知识服务对外能力：

- 生成发布快照
- 建立统一检索视图
- 只读已发布知识
- 支持回滚撤回
- 接入运营与审核后台
- 建立监控、告警、成本报表与审核积压看板

## 10. 首批实施范围建议

为了降低首轮风险，首批建议：

- 优先跑通 `20` 到 `50` 个高价值账号
- 第一批上线 `视频级摘要 + 剧情事件 + 人物关系 + 证据片段`
- 名台词候选、评论线索增强、图谱联动放在后续阶段

## 11. 仓库改动建议

首批应优先改动以下位置：

- `contracts/ingestion-format.md`
- `services/knowledge-pipeline/`
- `docs/architecture/`
- `apps/admin/`

## 12. 验收原则

第一期至少需要具备以下验证能力：

- adapter 解析单测
- 页面样本回归测试
- 契约测试
- 幂等与重试测试
- 发布快照回滚测试
- 从发现视频到发布知识的端到端烟测

## 13. 结论

Tongfuli 的视频能力应按“企业级来源治理系统”建设，而不是额外加一个独立爬虫脚本。对内保留完整第三方证据链、审核链与版本链，对外只发布 Tongfuli 自有结构化知识与总结，并统一纳入现有知识平台与发布快照体系。
